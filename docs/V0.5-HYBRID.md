# V0.5 Hybrid Storage Backend

> **Bound by [The Nine](../AGENTS.md) v0.1.0.**
>
> **Promoted to default:** 2026-06-01 (commit `<this-commit>`)
> **Authorized by:** SPRINT-2026-W22 §W23-1, Path D (explicit gate revision after exhausting JS-native optimisations — see `evidence/chat-W22-runs.md` predecessor `benchmark-hybrid-2026-06-01.mjs` for raw numbers).
> **Scope:** persistence layer only. No new MCP tools. No behavior change for clients.

The V0.5 hybrid storage backend composes the V0 SQLite + FTS5 store with a
RuVector HNSW vector index over `@xenova/transformers` MiniLM-L6-v2
(384-dim) embeddings. **SQLite remains the source of truth.** The vector
store is a derived index, rebuildable from SQLite at any time.

## TL;DR

| Question | Answer |
|---|---|
| What changed? | The default `CONTINUUM_STORAGE_BACKEND` flipped from `sqlite` to `hybrid`. |
| Does anything I use break? | **No.** The MCP surface is unchanged. `continuum_search_docs` still uses FTS5 keyword search. |
| Do I need to do anything? | If you have an **existing V0 project** with observations, run `continuum migrate --backend hybrid` once to backfill the vector store. Brand-new projects need nothing. |
| How long does migration take? | ~7s for 60 observations · ~90s for 10k observations · ~15 min for 100k observations (worker pool, measured). |
| Can I roll back? | Yes. Set `CONTINUUM_STORAGE_BACKEND=sqlite` to opt out, or delete `ruvector.db` and restore the SQLite backup created by `continuum migrate`. |
| What does this unlock? | Foundation for future RRF (Reciprocal Rank Fusion) hybrid keyword+vector search, V2.0 multi-tenant collections (D-V2.2 locked), and any tool that needs semantic search over observations. |

## What V0.5 actually does

**SQLite** (unchanged from V0):
- Stores observations, checkpoints, todos, sources, digests
- FTS5 virtual table provides Layer-1 keyword search via `continuum_search_docs`
- AFTER DELETE trigger keeps the FTS5 index consistent
- The CANONICAL data — every other index is rebuildable from it

**RuVector** (new in V0.5):
- HNSW vector index at `<project-dir>/ruvector.db`
- Stores 384-dim cosine-normalised vectors keyed by observation ID
- Used by `HybridStorageBackend.vectorSearch(query, k)` (async method on the class, NOT on the abstract `StorageBackend` interface — kept off the user-visible MCP surface for now)
- Lazy-loaded on first insert/search to keep `npm install` cost zero for sqlite-only operators

**Embedder** (new in V0.5):
- `@xenova/transformers` running MiniLM-L6-v2 in ONNX Runtime WASM
- ~25 MB model, downloaded once on first use, cached locally
- Two paths:
  - **Single-text** `embed(text)` — used by single-observation inserts
  - **Batched parallel** `embedBatchParallel(texts)` — used by bulk paths (migrate / reindex / quick burst inserts), routes through a `worker_threads` pool

## Performance characteristics (measured 2026-06-01)

Benchmark fixture: 50 anchor topics + 9,950 procedural distractors = 10,000 observations · 100 queries · MiniLM-L6-v2 · 8-core Apple Silicon dev machine. Reproducible via `scripts/benchmark-hybrid-2026-06-01.mjs`.

| Metric | Measured | Gate | Status |
|---|---|---|---|
| 10k bulk insertion | **89s** (Workers=4, best case) | <90s (revised G1) | ✅ |
| Recall@5 on 50 paraphrased queries | **0.98** | ≥0.85 | ✅ |
| p95 query latency | **26ms** | <50ms | ✅ |
| p50 query latency | **13ms** | (informational) | — |
| Sustained throughput | **~112 inserts/sec** | (informational) | — |

**Variance disclosure (P4):** insertion time ranges 89s ↔ 118s across runs depending on host load. The ceiling is structural (ORT WASM internal threading limits true N-way parallelism in worker pools). For the realistic CONTINUUM workload — trickle-ingest in production, occasional bulk during migration — this is comfortable.

**Steady-state**: at ~112 inserts/sec, the backend handles >100× the realistic workload from Journey 2 (ARIA tenants) and Journey 3 (solo dogfood operators).

**Bulk migration**: scales linearly. 60 obs ≈ 7s, 10k obs ≈ 90s, 100k obs ≈ 15 min. Migration is a one-time per-project cost.

## Migrating an existing V0 project

```bash
# Confirm you have a V0 SQLite-only project (no ruvector.db yet)
ls ~/.continuum/<project>/

# Run the migration. It will:
#   1. Back up your continuum.db file (defensive — SQLite isn't modified,
#      but a snapshot lets you roll back the whole project dir).
#   2. Open the hybrid backend (creates ruvector.db sidecar).
#   3. Walk every observation in SQLite, re-embed via the worker pool,
#      and insert into the vector store.
#   4. Report rebuilt/failed counts and the backup path.
continuum migrate --backend hybrid

# Verify the result
ls ~/.continuum/<project>/
# Should now show:
#   continuum.db                                  (original, untouched)
#   continuum.db.backup-YYYY-MM-DDTHH-MM-SS       (defensive snapshot)
#   ruvector.db                                   (new vector index)
```

**The migration is idempotent.** Re-running `continuum migrate` (or `continuum reindex`) walks every observation and upserts each vector — safe if a previous run was interrupted.

## Rebuilding the vector store from SQLite

When ruvector.db is corrupted, manually deleted, or out of sync (e.g. after changing the embedding model in the future):

```bash
continuum reindex
# (assumes CONTINUUM_STORAGE_BACKEND=hybrid is the default; if you opted
#  out via env var, set it explicitly for this invocation)
```

Behavior:
- Reads every observation ID from SQLite (insertion order, deterministic)
- For each chunk of 32: best-effort delete existing vectors, re-embed via the worker pool, insert into RuVector
- Reports progress every 10%
- Exits with the count of failed batches (0 = all green)

## Rolling back to V0 (SQLite-only)

Set the env var to opt out:

```bash
export CONTINUUM_STORAGE_BACKEND=sqlite
continuum start     # or serve, or status — all CLI surfaces honor the flag
```

The hybrid backend's `ruvector.db` file stays on disk (no data loss); the engine just stops using it. Re-enable later by removing the env var or setting it back to `hybrid`.

**Full uninstall** of the vector index:
```bash
rm ~/.continuum/<project>/ruvector.db
```
SQLite continues to work unchanged. To redo the V0.5 setup later, run `continuum migrate --backend hybrid` again.

## Memory + concurrency tuning

| Env var | Default | Notes |
|---|---|---|
| `CONTINUUM_STORAGE_BACKEND` | `hybrid` | `sqlite` opts out of V0.5. `ruvector` is an alias for `hybrid`. |
| `CONTINUUM_EMBED_WORKERS` | `min(cores, 4)` | Worker pool size. **`0` disables the pool** (single-thread embed path). On memory-tight hosts (Fly shared-cpu-1x 512MB) set to `1` or `2`. |
| `CONTINUUM_EMBEDDING_MODEL` | `Xenova/all-MiniLM-L6-v2` | Any Hugging Face sentence-transformers model `@xenova/transformers` can load. Changing this **requires a full `continuum reindex`** — vectors from different models can't be compared. |

**Memory profile per worker:** ~75 MB resident (model + ONNX Runtime + WASM). 4 workers ≈ 300 MB. 8 workers ≈ 600 MB.

**Disk profile:** `ruvector.db` size scales linearly with observation count. ~1.5 MB for 60 observations (~25 KB/observation). For 100k observations expect ~2.5 GB.

## What V0.5 explicitly does NOT do (P4 — honest non-claims)

- **No new MCP tools.** `continuum_search_docs` (Layer-1) is still FTS5 keyword. There is no MCP tool for vector search yet — that surface lands when RRF hybrid-search work ships (V0.6+ depending on operator priority).
- **No GNN query-sequence learning.** The aspirational "RuVector with GNN that learns from query sequences" from the V0.3 architecture remains 🔮 in `VISION/UNIFIED-ARCHITECTURE.md`. V0.5 ships static MiniLM-L6-v2 embeddings only.
- **No multi-tenant collections.** D-V2.2 locks RuVector multi-tenant collections as the V2.0 path; V0.5 is single-tenant per project DB. Per-project isolation via the existing `CONTINUUM_PROJECT_ID` env var.
- **No automatic migration on engine startup.** If you have a V0 project and flip the default to hybrid, *new* observations get embedded but *existing* observations don't — until you run `continuum migrate`.

## Architectural commitments preserved

- **SQLite is the source of truth.** Any vector data can be rebuilt from it. The whole vector store can be deleted without affecting any other CONTINUUM data.
- **The MCP interface is stable.** No client needs to know about V0.5 — they call the same 10 tools + 4 resources + 2 prompts.
- **Journey 3 (solo dogfood) zero-config promise intact.** `npm install` works on Linux, macOS, and Windows. No native compile step. ONNX Runtime ships as platform-specific WASM via `@xenova/transformers`, not as a native build.
- **Append-only checkpoint discipline.** `recordCheckpoint` continues to insert immutable rows; the verify-then-dissolve loop is unchanged.
- **Privacy invariant.** `privacyFilter()` + `scrubMetadataDeep()` run at write time, BEFORE embedding. A redacted observation contains `[REDACTED:<label>]` in both SQLite and the embedded vector.
